    CI must fail if bundle budgets are exceeded.
    10.2 Rendering Performance

The main overlay's text area must never trigger a layout reflow on every keystroke.
Word count updates must be debounced by 300ms — never on every keydown.
Backdrop blur must use CSS backdrop-filter — never JavaScript blur.
Animation: all animations use transform and opacity only (compositor-only properties).
Never animate width, height, top, left, background-color — these trigger layout/paint.
will-change: transform, opacity applied to the overlay container permanently.

10.3 Code Splitting

Onboarding flow and Settings panel are lazy-loaded — not included in the initial bundle.
React.lazy + Suspense for all non-critical panels.
Feature flag service and theme system are eagerly loaded (needed immediately).

typescript// App.tsx
const OnboardingFlow = React.lazy(() => import('./features/onboarding'));
const SettingsPanel = React.lazy(() => import('./features/settings'));

10.5 Asset Rules

All icons must be inline SVG or Tabler icon components — no PNG icons.
Self-hosted fonts only. Two font families maximum: UI font + mono font.
Font files: WOFF2 format only, font-display: swap.
No GIF or MP4 — all animations are CSS or JS-driven.
Image assets (if any) must pass through Squoosh compression before commit.

11.4 SaaS Product Principles

Every feature has a settings hook. If it's configurable, the hook reads from useSettingsStore first, then falls back to the default.
Settings persist immediately — no "Save" button. Use useDebouncedPersist to write 500ms after the last change.
Import/Export settings as a JSON file. This is the "profile sharing" feature for power users.
Keyboard-first. Every clickable control must be reachable via keyboard.
Onboarding is re-enterable. A "Reset & Replay Onboarding" option must exist in Advanced settings.

15. Error Handling Rules
    15.1 Frontend Error Handling

Every Tauri invoke call must be wrapped in a try/catch.
Errors must flow to the relevant Zustand store's error field, not be silently swallowed.
User-visible errors are displayed via the Toast component, never alert() or console.error.
Every error displayed to users must have an actionable message — never show raw Rust error strings.

typescript// Correct error handling pattern
async function showWindow() {
try {
await windowService.show();
} catch (err) {
const message = parseAppError(err) ?? 'Failed to show window. Try pressing the shortcut again.';
useAppStore.getState().setError(message);
}
}
15.2 Error Boundary

Every feature root component must be wrapped in an ErrorBoundary.
The overlay's ErrorBoundary fallback must be a minimal UI (just the ghost dot + a muted error message) — the overlay must never go fully blank on error.
Onboarding's ErrorBoundary fallback must offer a "Reset and restart" option.

16. Feature Flag System

Feature flags are stored in the Tauri plugin store under the feature_flags key.
All flags default to false (opt-in).
Flag evaluation happens in a single useFeatureFlags() hook — never evaluate a flag with an if on a raw config value elsewhere.
Flags are typed with the FeatureFlag type:

17. Theming Engine
    17.1 How It Works

Base tokens are loaded from tokens.css at startup (the CSS default).
On app load, ThemeProvider reads the persisted themeId from the store.
The preset's token overrides are written as inline styles on document.documentElement.
All CSS variables cascade from :root — every component inherits the theme automatically.
User customisations (opacity, accent colour) are written as individual token overrides on top of the preset.

17.2 Theme Application Function
typescript// features/theme/ThemeProvider.tsx
function applyTheme(preset: ThemePreset, userOverrides: Partial<DesignTokens>) {
const tokens = { ...preset.tokens, ...userOverrides };
const root = document.documentElement;
Object.entries(tokens).forEach(([key, value]) => {
root.style.setProperty(`--${key}`, value);
});
}
No re-renders. No CSS-in-JS. No class toggling. Pure CSS custom property mutation.

18. Accessibility Rules

All interactive elements must be keyboard-accessible.
Focus management: when the overlay appears, focus must move to the text area automatically.
When the overlay hides, focus must return to the element that had focus before.
ARIA labels required for all icon-only buttons.
Colour contrast minimum: 4.5:1 for body text, 3:1 for large text (WCAG AA).
High-contrast mode: when OS high-contrast is active, disable backdrop-filter and use solid backgrounds.
Reduce-motion: when prefers-reduced-motion is active, disable all entry/exit animations (use instant show/hide).

css@media (prefers-reduced-motion: reduce) {

19. Release & Build Rules
    19.1 Version Numbering
    Semantic versioning: MAJOR.MINOR.PATCH

MAJOR: Breaking change to settings schema or OS support
MINOR: New user-facing feature
PATCH: Bug fix or performance improvement

Version must be synchronised across package.json and tauri.conf.json. A script enforces this in pre-commit.
19.2 Build Pipeline

1. pnpm lint → Biome linting (zero warnings allowed)
2. pnpm typecheck → tsc --noEmit
3. pnpm test → Vitest (must pass 100%)
4. cargo test → Rust tests (must pass 100%)
5. cargo audit → No high/critical vulnerabilities
6. pnpm audit → No high/critical vulnerabilities
7. pnpm build → Vite production build
8. bundle size check → Fail if budget exceeded
9. tauri build → Sign + package for all 3 platforms
   All 9 steps must pass. Skipping any step requires a tracked exception with an expiry date.
