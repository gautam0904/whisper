4.5 Composition Over Inheritance

In React: compose components, never extend them.
In Rust: use traits + composition, not deep struct inheritance.
Use render props or compound component patterns for complex UI.

4.6 Command Query Separation (CQS)

Functions either change state (commands) or return data (queries). Never both.
Tauri commands that modify OS state return Result<(), Error>, never data.
Tauri commands that read state return Result<T, Error>, never modify anything.

5. Frontend Rules
   5.1 Component Rules

Functional components only. Zero class components.
Default export per file. Named exports for types and sub-components only.
Props interface always defined above the component, named [ComponentName]Props.
Components that receive more than 6 props must use a single props object — no long destructuring lists.
No React.FC — use plain function with typed props.

typescript// Correct component pattern
interface GhostDotProps {
active: boolean;
pulseInterval?: number;
}

export default function GhostDot({ active, pulseInterval = 1000 }: GhostDotProps) {
return <div className={...} />;
}
5.2 Hook Rules

All custom hooks start with use.
Hooks must be pure with respect to their declared dependencies.
Never call a hook conditionally.
Hooks that call Tauri APIs must handle both success and error states and expose loading, error, and data.
A hook file exports only one hook — no bundling multiple hooks in one file.

5.4 TypeScript Rules

strict: true in tsconfig.json — non-negotiable.
noImplicitAny: true — no exceptions.
as type assertions only in test files or when interfacing with untyped Tauri responses.
All Tauri command return types must have a corresponding Zod schema that validates the response at runtime.
unknown over any when a type truly cannot be known.
Enums are banned — use as const objects with a type helper.

// Correct
const PLATFORM = { WINDOWS: 'windows', MACOS: 'macos', LINUX: 'linux' } as const;
type Platform = typeof PLATFORM[keyof typeof PLATFORM];

6.2 Bootstrap Usage Rules
Use Bootstrap only for:

12-column grid (col-_ classes)
Flexbox utilities (d-flex, align-items-_, justify-content-_)
Display utilities (d-none, d-block)
Spacing utilities (mt-_, p-\*) only when the token system would be over-engineered

Never use Bootstrap for:

Colours (use our tokens)
Typography (use our tokens)
Buttons (use MUI Button or custom)
Modals/Dropdowns (use MUI equivalents)
Any JS-dependent Bootstrap component

Bootstrap must be imported as a partial — only the grid, utilities, and reboot. Never import all of Bootstrap.
typescript// vite.config.ts — import only what we need
// styles/bootstrap-partial.scss
@import 'bootstrap/scss/functions';
@import 'bootstrap/scss/variables';
@import 'bootstrap/scss/mixins';
@import 'bootstrap/scss/grid';
@import 'bootstrap/scss/utilities';
@import 'bootstrap/scss/reboot';
6.3 Tailwind Usage Rules

Configure Tailwind to use our CSS token values only — no default Tailwind colour palette.
Use Tailwind for one-off layout tweaks and state variants that don't justify a component style.
Tailwind classes in JSX are allowed; Tailwind classes in CSS @apply rules are allowed only in component-scoped CSS Modules.
Maximum 8 Tailwind classes on a single element — beyond that, extract to a CSS class.
Tailwind's backdrop-blur-_ and opacity-_ are the primary performance-safe utilities for the overlay.

6.4 MUI Usage Rules

Always use MUI's ThemeProvider with a theme built entirely from our CSS tokens.
Never hardcode colours in sx props — always reference the MUI theme, which in turn references our tokens.
Disable MUI's default global reset (disableGlobalStyles: true) — we use Bootstrap reboot.
Use MUI only for: Slider, Switch, Select, Tooltip, Snackbar (Toast), Menu, Dialog.
All other elements should be custom-built with Tailwind/CSS.
MUI components inside the main overlay must have disableRipple and disablePortal where applicable.

7. State Management
   7.1 State Layering
   Layer 1: Local State → useState, useReducer (ephemeral UI state)
   Layer 2: Feature State → Zustand store (shared within a feature)
   Layer 3: App State → Zustand store (cross-feature: visibility, theme, settings)
   Layer 4: Persistent State → @tauri-apps/plugin-store (survives restarts)
   7.2 Zustand Rules

One store file per feature/concern.
Stores export a typed hook: useOverlayStore, useSettingsStore, etc.
Stores must never import from React components.
Async actions inside stores use set + get pattern, not immer (keep bundle tiny).
Every store must have a reset() action for testing.

typescript// Correct Zustand store structure
interface OverlayState {
text: string;
isVisible: boolean;
wordCount: number;
}

interface OverlayActions {
setText(text: string): void;
setVisible(visible: boolean): void;
reset(): void;
}

const initialState: OverlayState = { text: '', isVisible: false, wordCount: 0 };

export const useOverlayStore = create<OverlayState & OverlayActions>((set) => ({
...initialState,
setText: (text) => set({ text, wordCount: countWords(text) }),
setVisible: (isVisible) => set({ isVisible }),
reset: () => set(initialState),
}));
7.3 What Must NOT Live in State

Window dimensions (query from Tauri API on demand)
System information (query from platform adapter on demand)
Computed values derivable from other state (use useMemo or Zustand get)

8.2 Error Handling in Rust

All errors must implement the AppError trait (defined in error.rs).
AppError must impl serde::Serialize for Tauri serialisation to the frontend.
Use ? operator freely — never .unwrap() in production code paths.
.expect() only in main.rs startup sequences where failure is unrecoverable.

rust// error.rs #[derive(Debug, thiserror::Error, serde::Serialize)]
pub enum AppError { #[error("Window operation failed: {0}")]
Window(String), #[error("Platform capability unavailable: {0}")]
PlatformUnsupported(String), #[error("Settings serialisation failed: {0}")]
Settings(String),
}

8.4 Tauri Capabilities (Security)
Every Tauri command must be explicitly listed in the capabilities JSON. Default deny-all. No wildcard permissions.
json// src-tauri/capabilities/default.json
{
"permissions": [
"core:window:allow-show",
"core:window:allow-hide",
"core:window:allow-set-focus",
"global-shortcut:allow-register",
"global-shortcut:allow-unregister",
"store:allow-set",
"store:allow-get"
]
}

9. Security Rules
   9.1 Tauri Security

CSP must be set in tauri.conf.json. No unsafe-inline for scripts. Ever.
dangerouslyUseHTMLString or any equivalent must never be used.
All user input that flows into a Tauri command must be validated with Zod on the frontend before the invoke call.
All Tauri command inputs must be validated with a Rust type and pattern-matched — never trust raw strings.
IPC payloads must be < 1 MB. Enforce with Zod's max() on string inputs.

9.2 Data Privacy

User notes are never sent to any network endpoint. The app is fully offline.
If a future version adds sync, all data must be E2E encrypted before leaving the device (key never leaves device).
No telemetry, analytics, or crash reporting without explicit user opt-in with a clear data policy screen.
@tauri-apps/plugin-store data is stored in the OS user data directory, never in the app bundle directory.

9.3 Dependency Security

Run pnpm audit and cargo audit in CI on every push.
Any high or critical severity vulnerability blocks the build.
Lock files (pnpm-lock.yaml, Cargo.lock) must always be committed.
Dependabot or Renovate must be configured for automated update PRs.
